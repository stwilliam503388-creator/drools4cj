## 🏗️ 核心模块设计

### 1. Fact 模块

**职责:** 定义事实对象的数据结构

**类图:**

```
                    ┌──────────────┐
                    │   <<abstract>>│
                    │    Fact      │
                    └──────┬───────┘
                           │
           ┌─────────────┼─────────────┐
           │             │             │
           ▼             ▼             ▼
    ┌─────────┐  ┌─────────┐  ┌─────────┐
    │OrderFact│  │ProductFact│  │Customer│
    └─────────┘  └─────────┘  │Fact    │
                               └─────────┘
```

**核心方法:**

```cangjie
public abstract class Fact {
    public var type: String
    public var id: String
    public abstract func matches(condition: Condition): Bool
}
```

**设计要点:**

- ✅ 抽象基类 - 所有事实继承此类
- ✅ 类型字段 - 支持类型索引
- ✅ ID 字段 - 唯一标识
- ✅ matches() - 条件匹配逻辑

### 2. Rule 模块

**职责:** 定义规则和条件

**类图:**

```
┌───────────┐     ┌──────────┐     ┌──────────────┐
│   Rule   │     │Condition │     │PropertyValue │
└─────┬─────┘     └─────┬─────┘     └──────┬───────┘
      │                 │                     │
      ▼                 ▼                     ▼
  ┌────────────────────────────────────────────────────────┐
│                        RuleBuilder                      │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ chain: RuleBuilder → RuleBuilder → RuleBuilder     │ │
│  │                                                  │ │
│  │ salience() → when() → when() → then() → build()    │ │
│  └──────────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────────┘
```

**核心方法:**

```cangjie
public class Rule {
    public func evaluate(fact: Fact): Bool {
        // 评估所有条件
    }

    public func execute() {
        // 执行所有动作
    }
}

public class RuleBuilder {
    public func when(field: String, operator: String, value: T): RuleBuilder
    public func then(action: Action): RuleBuilder
    public func build(): Rule
}
```

**设计要点:**

- ✅ 流式 API - Builder 模式
- ✅ 类型安全 - 编译时检查
- ✅ 链式调用 - 提供清晰语法
- ✅ 条件/动作分离 - 关注点分离

## 🔄 执行流程

### 规则执行完整流程

```
1. 用户调用 RuleEngine.fireAllRules()
       ↓
2. PatternMatcher.matchRules(rules)
       ├─ 遍历所有事实
       ├─ 对每个事实匹配所有规则
       └─ 返回匹配结果
       ↓
3. Agenda.addActivation()
       ├─ 为每个匹配创建 Activation
       ├─ 检查 no-loop 约束
       └─ 添加到待执行列表
       ↓
4. Agenda.sortActivations()
       ├─ 按 salience 排序
       ├─ 高优先级在前
       └─ 同优先级按时间戳
       ↓
5. Agenda.fireAll()
       ├─ 遍历激活列表
       ├─ 对每个未激活的激活:
       │   ├─ 通知监听器 onBeforeExecute
       │   ├─ 执行 rule.execute()
       │   └─ 通知监听器 onAfterExecute
       └─ 标记为已激活
       ↓
6. 输出统计信息
```

## 🎯 设计原则

### 1. SOLID 原则

- **S**ingle Responsibility - 每个类单一职责
- **O**pen/Closed Principle - 对扩展开放,对修改关闭
- **L**iskov Substitution Principle - 子类型可替换父类型
- **I**nterface Segregation Principle - 接口专一
- **D**ependency Inversion Principle - 依赖抽象

### 2. KISS 原则

- Keep It Simple, Stupid - 保持简单
- 避免过度设计
- 简洁的 API
- 清晰的代码

### 3. DRY 原则

- Don't Repeat Yourself - 避免重复
- 提取公共逻辑
- 复用代码模板

## 🔧 关键设计决策

### 1. 为什么选择 HashMap?

**决策:** Working Memory 使用 HashMap 存储事实

**原因:**
- ✅ O(1) 插入和查找
- ✅ 类型安全
- ✅ 内存效率高
- ✅ 支持快速迭代

**权衡:**
- ❌ 无序存储 - 但规则执行不需要顺序
- ❌ 内存开销 - 但在可接受范围内

### 2. 为什么选择简化 Rete?

**决策:** 使用线性匹配而非完整 Rete 网络

**原因:**
- ✅ 实现简单,易于理解
- ✅ 性能对小规模足够好
- ✅ 适合仓颉语言特性
- ✅ 可后续优化

**权衡:**
- ⚠️ 大规模性能下降
- ⚠️ 不支持复杂规则链

### 3. 为什么使用流式 API?

**决策:** RuleBuilder 提供 DSL 风格 API

**原因:**
- ✅ 类型安全
- ✅ IDE 友好(自动补全)
- ✅ 链式调用易读
- ✅ 比字符串 DSL 更安全

**示例对比:**

```cangjie
// Drools4Cj - 流式 API
engine.addRule(
    RuleBuilder("规则")
        .when("amount", ">", PropertyValue(1000.0))
        .then({ println("ok") })
        .build()
)

// vs Drools DRL 文件
rule "规则"
    when
        $order: Order(amount > 1000)
    then
        System.out.println("ok");
end
```

## 📊 性能考虑

### 时间复杂度

| 操作 | 复杂度 | 说明 |
|------|--------|------|
| 插入事实 | O(1) | HashMap 插入 |
| 查找事实 | O(1) | HashMap 查找 |
| 按类型查找 | O(1) | HashMap 查找 + 遍历 |
| 模式匹配 | O(n×m) | n个事实,m个规则 |
| 排序激活 | O(k log k) | k个激活 |

### 空间复杂度

| 组件 | 空间复杂度 | 说明 |
|------|-----------|------|
| facts | O(n) | n 个事实 |
| typeIndex | O(n) | n 种类型 |
| activations | O(k) | k 个激活 |
| 总计 | O(n) | 线性增长 |
